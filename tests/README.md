# 채점 로직(gradeProblem.ts) 테스트 가이드

AI 채점(`src/lib/gemini/gradeProblem.ts`)은 자동 유닛테스트가 어렵다 — 결과가 실제 Gemini 호출에 달려있고, "정답"이 이산적인 pass/fail이 아니라 점수·근거의 타당성이라 어서션을 세심하게 설계해야 한다. 이 문서는 지금까지 이 파일들로 무엇을, 왜, 어떻게 검증했는지와 다음에 새 버그를 의심할 때 어떤 실험을 먼저 시도하면 되는지 정리한 것이다.

**사용 모델**: `gemini-3.1-flash-lite`(`thinkingConfig: { thinkingLevel: 'minimal' }`, 2026-07-21 전환 완료, Slice 1 반영됨).

> **모델 변경 이력**: 2026-07-12에 한 번 `gemini-3.1-flash-lite`로 올렸다가(문제1 판례 법리 관대화 3건 중 1건이 정상화되는 효과는 있었으나) 실측 비용이 약 5배 뛴다고 판단해 `gemini-2.5-flash-lite`로 되돌렸다. 2026-07-21에 `thinkingLevel=minimal` 설정으로 재실측한 결과 이 "5배" 기록이 재현되지 않아(아래 참고) 다시 3.1-flash-lite로 전환했다. 물음단위 독립 호출 아키텍처(버그#10)는 비용 증가폭이 훨씬 작아서(+약 77%, 구조적 버그 수정이기도 함) 이번 전환과 무관하게 그대로 유지한다.
>
> ✅ **Slice 0(비용 상충 해소) 완료 → Slice 1(모델 교체) 완료.** 최초 재조사(문제51 물음2, 1문항×5회 반복)에서 이미 1.5~1.8배가 나왔었고, 이후 `verify-grading.ts`의 6개 픽스처(문제1·9·46·51·22·57, 총 14개 물음)를 `gemini-3.1-flash-lite`+`thinkingLevel=minimal`로 전량 재현(각 1회)한 결과 **비용 배율이 1.57~1.76배 사이(종합 1.67배)로 문제 종류와 무관하게 일관되게 수렴**했고, **지연은 종합 0.28배(약 3.6배 빠름)** 로 역시 일관됐다. "5배"는 이 설정(thinkingLevel=minimal)으로는 재현되지 않는다 — 2026-07-12 실험은 아마 `thinkingLevel=high`급(dynamic thinking, 물음당 최대 122~127초 관측)으로 돌았을 가능성이 가장 유력하다. 다만 이 6문제 스윕은 프로덕션의 `normalizeScoresAgainstRubrics`(배점 캡핑) 없이 모델의 원본 자체 집계를 그대로 썼기 때문에(문제51·57 일부 물음에서 3.1-minimal이 배점을 초과해 자체 집계하는 사례 관측 — 버그#3과 동일 계열, 프로덕션에서는 캡으로 무해함) **점수 품질 비교로는 쓰지 말 것** — 품질 비교는 아래 `--problem51 --overgeneralized`(결론 역전 트랩) 쪽이 근거가 더 탄탄하다.
>
> **Slice 1 배포 후 회귀 리뷰(2026-07-21)에서 발견된 잔여 리스크**: `--problem51 --overgeneralized`의 집계 임계값(40% 미만)은 통과하지만, 물음1 "결론" 기준 하나가 학생의 역전된 결론 문장을 근거로 met 처리되는 사례가 재현됨(1점이라 집계엔 안 잡힘) — evidenceQuote 검증이 "답안에 실존하는가"만 확인하고 "요구된 결론과 같은 방향인가"는 확인하지 않는 기존 한계(아래 "알려진 한계" 참고)가 새 모델에서도 그대로 재현된 것. 이를 잡기 위해 `--overgeneralized` 모드에 "결론" 기준 개별 어서션을 추가함(아래 명령 실행 시 자동 확인).

## 보조 스크립트 (회귀 스위트 외)

- **`audit-problems.ts`**, **`fetch-problem-detail.ts`** — 2026-07-13에 `cta_uploader/`로
  이동했다(문제 출제/업로드 도구인 cta_uploader와 함께 관리하기 위함 — 이 앱 자체의 git
  이력에는 잡히지 않도록 `cta_uploader/`는 `.gitignore` 처리됨). `../src/lib/gemini/gradeProblem`을
  상대경로로 import하므로 CTA_tax_law 바로 아래(`cta_uploader/`)에 있어야만 동작한다. 실행은
  `cta_uploader/`에서 `npx -y tsx --env-file=../.env.local audit-problems.ts ...` 형태(env
  파일이 한 단계 위를 가리키는 점 주의). 용도와 플래그는 그대로: `audit-problems.ts`는
  각 문제의 루브릭별 `example_answer_text`를 이어붙인 모범답안을 실채점기에 넣고 만점이 안
  나오는 루브릭(데이터 결함 신호)을 찾는다(`--id N`, `--from/--to`, `--file draft.json` —
  create-problem/review-problem 스킬의 dry-run 게이트). 2026-07-13 전수(48문제) 실행 결과:
  문제8 물음1 "기준면적 이내 과세구분" 1/2, 문제56 물음2 "판례의 논거" 2/4 — 모범답안이 자기
  설명의 요소를 다 담지 못한 2건 발견(수정 필요), 나머지 46문제 전부 만점 통과.
  `fetch-problem-detail.ts`는 특정 문제의 전체 데이터(루브릭 설명·모범답안·키워드 포함)를
  Supabase에서 조회한다 (`--id N`, `--json`). 문제 데이터는 레포에 없고 DB에만 있음.

- **`fetch-problems.ts`** → **`generate_tests.ts`** → **`verify-extra.ts`**: 이 세 개는 하나의
  작은 파이프라인이다. `fetch-problems.ts`는 Supabase의 전체 문제 id·제목을 나열해(API 불필요한
  탐색용) 어떤 문제를 다음에 다룰지 고르는 데 쓴다. `generate_tests.ts`는 `TARGET_IDS` 배열에
  적힌 문제들을 `cta_uploader/data/problem_N.json`에서 읽어 `verify-extra.ts`를 자동 생성한다
  (`npx tsx tests/generate_tests.ts`로 재생성). `verify-extra.ts`는 `verify-grading.ts`의 6개
  정식 픽스처와 별도로, DB의 다른 미시행 기출문제(현재 2·12·52)를 강한 답안 하나로만 채점해
  실채점기가 최소한 정상 동작하는지(점수·API 오류 유무)만 확인하는 **넓고 얕은** 스윕이다 —
  incomplete/half/ambiguous 등 답안 유형별 검증이나 산술·근거 어서션은 없다(`verify-grading.ts`처럼
  정밀 검증이 필요해지면 그 문제를 정식 픽스처로 승격할 것 — 문제22가 바로 이 경로를 거쳤다).

  ```bash
  npx -y tsx --env-file=.env.local tests/verify-extra.ts
  ```

  2026-07-21에 문제22 데이터를 `verify-extra.ts`와 `generate_tests.ts`의 `TARGET_IDS`에서 제거했다
  — `verify-grading.ts`의 `PROBLEM22_FIXTURE`(incomplete 모드·산술/근거 어서션까지 갖춘 정식
  픽스처)로 완전히 대체되어 중복이었기 때문. 이 세 파일은 `cta_uploader/data/problem_N.json`을
  그대로 파싱해 만들어져 `ProblemWithDetails`와 완전히 같은 타입 모양은 아니다(예: `id` 필드 누락,
  `case_text_full`이 배열인 곳 등) — `tsc`로 직접 확인하면 타입 에러가 여러 건 나오지만, `tsx`가
  타입을 지우고 실행만 하므로 런타임에는 문제없이 동작한다(기존부터 그래왔던 특성이며 이번 정리로
  새로 생긴 문제가 아님).

문제 1(재산가치 증가이익의 증여, 상속세및증여세법), 문제 9(면세전용과 공통매입세액 재계산, 부가가치세법), 문제 46(공장·본사 지방이전 세액특례, 조세특례제한법) 세 개의 서로 다른 과목·구조 픽스처를 내장하고 있다. 문제를 하나만 쓰면 그 문제의 특정 문구에 과적합된 수정을 놓칠 수 있어서 처음부터 여러 개를 유지했다. 문제 9는 물음당 루브릭이 최대 6개(문제1·46은 2~3개)라 루브릭 개수가 많을 때도 산술·근거검증 로직이 깨지지 않는지 확인하는 용도를 겸한다.

2026-07-21에 세 개를 더 추가했다 — 문제 51(중복조사 금지 예외와 조사 한계, 국세기본법: 한때 내용 변경 예정으로 제외되었으나 이제 프로덕션에서 안정적으로 쓰이고 있어 재도입), 문제 22(과세예고통지 생략과 절차적 권리침해, 국세기본법: `investigate-q22-conclusion.ts` 진단 데이터를 회귀 스위트로 승격), 문제 57(유사매매사례가액의 시가 인정, 상속세및증여세법: 실제 프로덕션 로그의 과소평가 의심 사례를 strong 모드 회귀로 승격). 아래 "새 픽스처 상세(2026-07-21)" 절 참고.

## 새 픽스처 상세 (2026-07-21 추가)

세 문제 모두 문제51 물음2 실제 채점 리뷰·문제57 실제 채점 리뷰·`gemini-3.1-flash-lite` 마이그레이션 검토 과정에서 나온 실제 사례를 회귀 스위트로 승격한 것이다. 세 개 다 실제 Gemini 호출로 검증 완료(아래는 그 실행 결과).

- **문제51 `--overgeneralized`(결론 역전 트랩)**: 물음1 답안은 실제 프로덕션 로그(attempt `37a3ea42...`)에서 그대로 가져온 실제 학생 답안 — 판례 법리는 그럴듯하게 서술하면서 사실관계 적용에서 결론을 정반대로 뒤집었다(모범답안: 재조사 적법·주장 부당 / 학생: 재조사 위법·주장 타당). 실행 결과 총점 4/15(27%)로 40% 미만 기준 PASS — 2.5-flash-lite가 이 답안에 실제로 과도한 점수(원래 로그에서 4.5/8, 약 56%)를 준 것과 대비된다. 이 모드는 향후 프롬프트·모델 변경이 이 트랩을 계속 잡아내는지 확인하는 회귀 기준점이다.
- **문제51 `--problem51`(strong)**: 15/15(100%) 만점, 전 루브릭 met. 단, 물음1 루브릭 배점 합(4+4+3+1=12)이 물음 배점(8)을 초과하고 물음2도 루브릭 배점 합(8)이 물음 배점(7)을 초과하는 **기존 DB 데이터 결함**이 있다(`normalizeScoresAgainstRubrics`가 물음 배점으로 캡핑하므로 채점 자체는 정상 동작하지만, DB의 `max_score` 재검토가 필요 — 코드 버그 아님). 실행 시 `물음 1/2 루브릭 합산 === 물음 점수` 체크가 FAIL로 뜨는 것은 **상시 알려진 실패**이며 회귀 신호가 아니다.
- **문제22 `--problem22`(strong)**: 세 루브릭의 `example_answer_text`를 결론→제도의 취지→판례 법리 포섭 순으로 그대로 이어붙인 답안으로 12/12 만점 확인(20/20 총점) — 표현이 모범답안과 동일해도(규칙 5) 정당하게 만점이 나오는지 보는 리트머스 시험지.
- **문제57 `--problem57`(strong)**: 물음2 답안은 실제 프로덕션 로그(attempt `1881e5dc...`)에서 그대로 가져온 실제 학생 답안. 실제 프로덕션 채점은 15/20(75%)이었고, 재현 실행에서는 16/20(80%)로 매번 다르게 나왔다(temperature=0인데도 재현 실행마다 점수가 바뀜 — 아래 "관찰" 참고). **총점 84% 미만은 상시 알려진 실패로 유지한다** — "기간 요건의 예외적 확장"·"절차적 요건" 두 기준 모두 답안에 정확하고 완전한 근거 문장이 이미 있음에도(모범답안보다 더 정밀한 "전 2년부터 후 법정결정기한까지"라는 표현까지 사용) 실행마다 2~4/4 사이를 오간 것으로 보아 과소평가 의심이 있으나, 이는 제 도메인 판단이며 100% 확신은 아니다.

### 관찰: 문제51·57에서 재확인된 2.5-flash-lite의 실행 간 변동성

문제57 `--problem57`을 정확히 동일한 코드·답안으로 두 번 실행했더니 15/20 → 16/20으로, 물음1 "원칙적 평가기간"이 4/4(met) → 2/4(partially_met)로 바뀌는 등 세부 배점 분포 자체가 달라졌다. temperature=0으로 설정되어 있음에도 Gemini API가 완전한 결정성을 보장하지 않는다는 뜻이며, 이 스위트의 실행 결과(특히 84% 경계선에 걸친 문제57 같은 사례)를 "1회 실행 = 정답"으로 해석하지 말고 여러 번 반복해 분포로 판단해야 한다.

모드별로 하는 일:

| 모드 | 답안 | 확인하는 것 |
|---|---|---|
| (기본) strong | 루브릭 내용을 실제로 담은 충실한 답안 | 총점이 만점 근처(≥84%)인지, 실존 표현을 허위로 "누락"이라 지적하지 않는지 |
| `--incomplete` | 핵심 요건이 빠진 부실한 답안 | 관대하게 채점되지 않는지(총점 <60%) — 엄격함을 잃지 않았는지 확인 |
| `--half` | 물음마다 루브릭 일부만 채우고 나머지는 완전히 비운 답안 | 채운 루브릭은 고득점(≥70%), 비운 루브릭은 저득점(≤30%)인지 — 점수가 실제 정답 비율에 비례하는지 (루브릭 단위의 이진 포함/누락) |
| `--ambiguous` | STRONG과 논리·내용은 동일하지만 정확한 법률 용어 대신 구어체로 에둘러 쓰고 결론도 완곡하게 표현한 답안 | 정확한 키워드가 없다는 이유만으로 루브릭이 부당하게 0점(unmet) 처리되지 않는지, 총점이 만점의 60% 이상 유지되는지 |
| `--overgeneralized` | 막연하거나 자신 없는 게 아니라, 자신감 있는 어조로 그럴듯한 일반 원칙을 내세우되 이 사안의 구체적 요건·예외를 무시하거나 뒤집어서 결론 자체가 틀린 답안 | 권위 있어 보이는 말투·법률 용어에 속아 관대하게 채점되지 않는지 (총점 <40%, `--incomplete`의 60%보다 더 엄격한 기준) |
| `--partial` | STRONG과 동일하되, 각 물음마다 여러 요건을 담은 루브릭 하나씩을 골라 그중 한 요건을 통째로 빼고 서술 (같은 물음의 다른 루브릭은 STRONG 그대로) | 그 루브릭들이 각각 `status===partially_met`이면서 0점도 만점도 아닌 진짜 부분 점수를 받는지, 같은 물음의 무관한 다른 루브릭까지 덩달아 깎이지 않는지 (루브릭 **내부** 요건 일부 충족 — `--half`의 세분화 버전) |
| `--merged` | STRONG과 논리·내용은 완전히 동일하되, 물음마다 여러 루브릭에 걸친 서술을 마침표로 끊지 않고 쉼표·연결어미("~하고", "~하며", "~는데")로만 이어 하나의 긴 문장으로 압축 | 문장 경계가 사라져도 각 루브릭의 근거를 놓치거나 엉뚱한 루브릭에 잘못 귀속시키지 않고 물음당 2~6개 루브릭 각각을 개별적으로 정확히 채점하는지(각 루브릭 ≥80%, 총점 ≥84%) — 다른 모드가 답안의 "내용"을 바꾸는 것과 달리 이 모드는 오직 "문장 구조"만 바꾼다 |

`--half`는 세 문제 모두에서 검증한다: 문제1(물음당 루브릭 2~3개 중 일부만 채움), 문제9(물음1은 "취지" 2개만 채우고 "산정방법"은 비움, 물음2는 6개 중 앞 3개인 "요건"만 채우고 "시기"·"배제사유" 3개는 비움), 문제46(물음마다 뒤/가운데/앞 루브릭을 번갈아 비워 위치 편향 없이 검증). 세 문제 모두 총점이 만점의 30~65%(대략 절반) 범위에 들어와야 통과 — 실제 실행 결과 문제1 12/25(48%), 문제9 10/20(50%), 문제46 14/25(56%)로 전부 통과했다.

`--merged`도 세 문제 모두에서 검증했다 — 세 문제 모두 만점(문제1 25/25, 문제9 20/20, 문제46 25/25), 물음 2개 루브릭부터 문제9 물음2의 6개 루브릭까지 전부 개별 만점, evidenceQuote도 각 clause 경계에 맞춰 정확히 분리 인용됨. 문장 부호로 답을 구획하지 않는 학생 답안(예: 개조식 없이 줄글로 몰아 쓰는 경우)에도 채점기가 흔들리지 않는다는 근거.

`--partial`은 각 문제의 **모든 물음**에서 하나씩(문제1: 물음1·2·3, 문제9: 물음1·2, 문제46: 물음1·2·3 — 총 8개 대상 루브릭) 검증한다. 최초 도입 시(문제 전체를 한 번에 채점하던 구버전 `gradeProblem.ts`) 8개 중 3개가 요건이 통째로 빠졌는데도 `met`+만점으로 FAIL했다. 원인을 일회성 진단 스크립트(해당 물음 하나만 떼어낸 미니 문제로 단독 채점 후 묶음 채점 결과와 대조, 이제는 삭제됨)로 추적한 결과 두 가지 다른 원인이 섞여 있었다:

1. **문제1 물음1·2·3 — 단독 채점해도 그대로 재현되는, 배치와 무관한 진짜 과다 관대화.** 시스템 프롬프트 규칙 8이 명시적으로 요구하는 시기 요건(5년 이내)·법리 요건(판례 법리 2요소 중 1개)이 완전히 빠졌는데도 감점하지 않은, 배치 구조와 무관한 채점 판단 문제. **아래 아키텍처 변경으로도 해결되지 않으며, 별도로 아직 미해결 상태다** (`tests/test.md` 참고).
2. **문제9 물음1, 문제46 물음3 — 다물음 동시채점 교차오염(배치 특유의 아티팩트).** 해당 물음만 단독으로 채점하면 정상적으로 partially_met이 나오는데, 다른 물음과 묶어 채점하면 그 내용에 영향을 받아 만점으로 튀어 올랐다. 기존 버그#9("정확한 근거를 인용하고도 unmet+0점, 단독 채점 시엔 정상")와 같은 계열이지만 방향이 반대(부족한 답안이 깎이는 게 아니라 옆 물음 덕에 끌어올려짐)이며, 시스템 프롬프트 규칙 13(기준별 독립 판단) 위반 사례다.

**2번(교차오염)을 해결하기 위해 `gradeProblem.ts`를 물음 단위 독립 호출 구조로 리팩터링했다** (문제 전체를 한 번의 Gemini 호출로 채점하던 방식 → 물음마다 별도 호출로 병렬(`Promise.all`) 채점 후 결과를 병합하고, 총평만 물음별 결과를 요약해 마지막에 한 번 더 합성). 리팩터링 후 재검증 결과: 문제9 물음1, 문제46 물음3 모두 정상적으로 partially_met으로 채점되어 **PASS**. 문제1 물음1·2·3은 예측대로 met+만점 그대로 남아 **여전히 FAIL**(배치 문제가 아니었다는 가설이 확인됨). 부수적으로 문제46 `--ambiguous`(아래 참고)에서도 예전에 실패하던 케이스가 함께 해결되었다.

물음 단위 호출은 시스템프롬프트+사실관계+쟁점(문제당 고정 오버헤드) 를 물음 개수만큼 중복 전송해 입력 토큰이 문제당 +42~65% 늘지만(`countTokens`로 실측), Gemini 2.5 Flash-Lite 요금($0.10/1M 입력, $0.40/1M 출력) 기준으로 환산하면 채점 1회당 비용 증가는 $0.001 미만으로 무시할 만한 수준이다. 다만 물음 개수만큼 **동시에** Gemini를 호출하므로, API가 일시 과부하(503)일 때 전체 채점 요청이 실패할 확률은 호출 수에 비례해 커진다(실측: 회귀 검증 중 물음 3개짜리 문제에서 503이 몇 차례 발생 — 재시도 로직 자체는 정상 동작했고 재실행하면 통과했다).

`--ambiguous` 실행 결과: 문제1 25/25(100%), 문제9 20/20(100%)로 전부 통과한다. 문제46도 리팩터링 이후 23.5/25(94%)로 개별 체크까지 전부 PASS — 예전에 실패하던 물음3 "감면대상 소득비율 산출"(답안에 "40/60=66.6%(2/3)"라는 산출식이 있는데도 unmet(0점) 처리되던 사례, `tests/test.md` 참고)이 물음 단위 독립 채점으로 전환되며 함께 정상화되었다.

`--ambiguous` 답안을 작성할 때 주의할 점 하나: 수치로 정해진 요건("면세비율 차이 5% 이상")을 완곡하게 쓴다고 "다섯 퍼센트 **정도** 이상"처럼 "정도"를 붙이면 안 된다 — 이는 단순한 구어체 표현이 아니라 정확한 수치 요건에 불확실성을 더해 실제로 더 약한(틀린) 주장이 되기 때문이다. 실제로 문제9 물음2 "재계산 요건 3"에 "정도"를 넣었더니 0/2(unmet)로 처리됐고, "정도"를 빼자(그 외 문장 구조는 그대로 두고) 1/2(partially_met)로 나왔다 — 채점 로직의 버그가 아니라 테스트 답안 설계 실수였다. 수치 요건은 "정도·대략·쯤" 같은 헤지 없이 정확한 숫자를 그대로 쓰되, 그 앞뒤 문장만 구어체로 풀어써야 "논리는 맞지만 표현만 애매한" 경우를 제대로 검증한다.

`--overgeneralized`는 `--incomplete`와 다르다: incomplete는 막연하고 자신 없는 부실한 답안(예: "~인 것 같다")인 반면, overgeneralized는 그럴듯한 일반 원칙("이 제도의 취지는 ~이므로", "대법원도 ~폭넓게 인정해 왔다")을 자신 있게 내세우면서 정작 이 사안에 적용되는 구체적 요건·예외·수치 기준을 무시하거나 결론을 통째로 뒤집는 답안이다. 채점 모델이 권위 있어 보이는 어조나 법률 용어 사용 자체에 속아 관대하게 점수를 주는지를 별도로 검증하기 위해 incomplete보다 더 엄격한 기준(40% 미만)을 쓴다. 실행 결과 세 문제 모두 통과했다(문제1 3/25=12%, 문제9 0/20=0%, 문제46 0/25=0%) — 특히 문제9(제도 자체를 부정한 답안)와 문제46(수도권 안/밖 구분 등 구체적 요건을 무시한 답안)는 완전히 0점 처리되어, 자신감 있는 어조에 흔들리지 않고 결론의 정오를 정확히 가려냈다.

**모든 모드에서 공통으로 도는 어서션** (문제·답안 내용과 무관하게 항상 참이어야 함):

- Σ루브릭 점수 === 물음 점수, Σ물음 점수 === 총점 (산술 일관성)
- `status !== 'unmet' && awardedScore > 0`인 모든 루브릭에서 `evidenceQuote`가 실제로 해당 물음 답안에 존재함 (근거 없이 점수 준 게 아닌지)
- `awardedScore === maxScore`(만점)인 모든 루브릭에서 `status === 'met'` (버그#11 회귀 방지 — partially_met 라벨인 채로 만점이 나오지 않는지)
- 미충족(unmet)·부분충족(partially_met) 기준이 있는 물음마다, 피드백에 `hasActionableRevisionInstruction()`이 인식하는 실행 가능한 지시문(추가/수정/명시/구체화/보완/작성/제시/서술/설명 + 하세요/하십시오/바랍니다/필요합니다/해야/하여야/좋겠습니다 등 어미)이 포함되어 있는지 — 2026-07-21에 "~해야 합니다"·"~하면 좋겠습니다" 어미를 놓치던 오탐(false negative) 두 건을 발견해 어미 목록을 보강했다(문제51/22/57 회귀 추가 과정에서 드러남).

이 공통 어서션은 `gradeProblem.ts` 내부 함수(`findPhantomEvidence` 등)를 import해서 재사용하지 않고 스크립트 자체 로직으로 **독립 재구현**했다 — 수정 코드와 검증 코드가 같은 버그를 공유하면 그 버그를 영원히 못 잡기 때문.

일회성 원인 규명이 필요할 때는 `investigate-q22-conclusion.ts`(문제22 물음2 "결론" 감점 사례 진단)의 구조를 템플릿으로 복사해서 쓴다: "채점이 이상하게 나온다"는 관찰만으로는 그게 (a) 모델의 정당한 엄격함인지 (b) 실제 버그인지 구분이 안 되므로, 아래 "새 버그 의심될 때 실험 순서"의 4단계를 그대로 적용해 좁혀나간다. 정식 회귀 스위트가 아니라 진단이 끝나면 지워도 되는 일회성 스크립트다(과거 문제51 물음3 진단용 `investigate-q3-strictness.ts`는 문제51 픽스처 제외와 함께 삭제됨 — 결과는 `tests/test.md` 참고).

## 새 버그(또는 "버그 같은데 확신이 안 서는 현상")를 의심할 때 실험 순서

이상 채점 사례를 하나 발견했을 때, 아래 순서로 실험하면 원인을 빠르게 좁힐 수 있다. 실제로 이 순서로 과거 문제51 물음3 사례의 원인(다물음 동시채점 시 주의력 분산)을 확정한 바 있다.

1. **재현성 확인** — 같은 문제·답안으로 3회 반복 실행(temperature 0이라 거의 결정론적). 매번 같은 결과면 "우연한 흔들림"이 아니라 코드/프롬프트에 박힌 systematic 문제.

2. **모범답안 자체 테스트** — 루브릭의 `example_answer_text`를 그대로 제출해도 같은 문제가 재현되는가? 재현되면 **학생 답안 문제가 아니라 채점 기준·프롬프트 자체의 버그**. 재현 안 되면 학생 답안의 특정 표현 문제이거나 다음 단계(단독 채점)로.

3. **단독 채점 vs 묶음 채점 비교** — 문제가 있던 그 물음만 떼어내(다른 물음 없이) 같은 답안으로 채점했을 때도 재현되는가? 여기서 결과가 달라지면(묶으면 실패, 단독이면 성공 등) **다물음 동시 채점 특유의 교차 오염** — 과거 실제로 발견된 적 있는 패턴. `gradeProblem()`은 `ProblemWithDetails.cta_subquestion` 배열 길이에 무관하게 동작하므로, 물음 1개짜리 미니 `problem` 객체를 즉석에서 만들어 비교하면 된다 (`investigate-q22-conclusion.ts`의 미니 `problem` 구성 패턴 참고).

4. **답안 강화/변형 테스트** — 의심되는 문구를 명시적으로 추가하거나 표현을 바꿨을 때 점수가 반응하는가? 반응하면(강화하니 점수가 오름) 어느 정도는 정당한 엄격함일 수 있음. 원본과 강화판이 "단독으로는 둘 다 만점, 묶으면 둘 다 0점" 식으로 답안 내용과 무관하게 같이 움직이면 답안 문제가 아니라 구조적 문제라는 뜻.

이 4가지를 다 실행해도 애매하면, `_diagnostics` 필드(교정 재시도가 실제로 발동했는지·어떤 모순이었는지)와 `console.warn` 로그(`[grading] ...`)를 같이 확인한다 — `gradeProblem.ts`는 재시도/강제 0점 처리마다 로그를 남긴다.

## 다른 문제로 픽스처 추가하는 법

`verify-grading.ts`의 `ProblemFixture` 인터페이스를 따른다:

```ts
interface ProblemFixture {
    problem: ProblemWithDetails       // cta_uploader/data/problem_N.json 또는 add_problem_N.py에서 그대로 복사
    label: string
    strongAnswers: Record<number, string>   // 물음 번호 → 충실한 답안
    incompleteAnswers: Record<number, string>
    halfAnswers?: Record<number, string>            // 선택
    halfCoverage?: Record<number, { covered: string[]; omitted: string[] }>  // half 답안 작성 시 함께 정의
}
```

문제 데이터는 실제 DB에 올라간 값과 반드시 일치시킨다(`cta_uploader/data/problem_N.json`이 원본). `keywords_json`은 실제로는 문자열 배열이지만 타입 정의가 `Record<string, unknown>`이라 스크립트 상단의 `kw()` 헬퍼로 캐스팅해서 넣는다.

## 미해결로 남은 이슈

- **`--partial` 모드의 "과다 관대화" — 배치와 무관한 진짜 문제, 문제1에 국한되지 않음**: 대상 루브릭에서 요건 하나를 통째로 빼도 met+만점을 주는 현상. 원래 문제1(물음1 "재산가치 증가사유 발생 요건", 물음2·물음3 "판례 법리")에서만 발견됐었으나, **2026-07-21 `--all` 전체 실행(gemini-3.1-flash-lite 전환 이후)에서 문제9("공급가액 산정방법")·문제46("공장이전 과세특례 요건")에서도 동일 패턴이 재현됨을 확인** — 문제1 특유의 현상이 아니라 `--partial` 모드가 겨냥하는 "법리 구성요소·부수적 처리귀결형 루브릭"에 걸쳐 나타나는 일반적 한계로 범위를 넓혀 이해해야 한다. 해당 물음만 완전히 단독으로 채점해도(버그#10 수정 이후에도) 요건이 통째로 빠진 답안에 met+만점을 준다. 시스템 프롬프트 규칙 8이 명시적으로 요구하는 요건 누락인데도 감점하지 않은, 정당화하기 어려운 판단으로 코드 버그#10과는 원인이 다르다(배치 문제가 아님).
  - **모델 업그레이드 실험 1차** (2026-07-12, gemini-2.5-flash-lite 기준): `gemini-3.1-flash-lite`로 교체하면 문제1 물음1은 정상화됨(프롬프트 변경 없이). 그러나 채점당 비용 약 5배 대비 개선이 3건 중 1건에 그쳐 되돌림 — 수익화 목표(월 구독자 100명+) 하에서 비용 효율 우선.
  - **모델 업그레이드 실험 2차** (2026-07-21, 실제 마이그레이션 완료 후 `--all`로 재확인): `gemini-3.1-flash-lite`(thinkingLevel=minimal)로 전환된 지금도 문제1 물음1이 **여전히 met+만점**으로 재현됨 — 1차 실험의 "정상화됨" 관측과 상충한다. 1차 실험이 다른 thinkingConfig(예: thinkingLevel=high급)로 돌았거나 단순 표본 변동(1회 관측)이었을 가능성이 있다. 즉 3.1-flash-lite로의 전환 자체가 이 이슈를 해결해주지는 않는다는 것이 현재까지 더 신뢰도 높은 결론이다.
  - **프롬프트 교정 실험 4회, 전부 실패 후 원복** (2026-07-12, 2.5-flash-lite 기준): ① 규칙 8에 요소 완전성 요구+실패 사례 예시 추가 → 효과 없음(예시가 실제 케이스와 거의 동일한데도 무시됨). ② 규칙 7을 판정 절차(요소 분해→요소별 확인→전부 있어야 met)로 재정의 → 효과 없음. ③ 응답 스키마 status 필드 설명에 판정 기준 명시 → 물음3이 met→**unmet(0/4)**으로 과잉 반응(부분 점수가 아니라 0점 — 답안에 실존하는 요소까지 부정), 나머지 불변. ④ 스키마에 `elementCheck` 필드를 추가해 status 앞에 요소별 있음/없음 점검을 출력하도록 구조적 강제 → 물음2는 고쳐졌으나(1.5/3 partially_met), **건드리지 않은 완전한 답안의 "수증자 요건"(OR형 나열 루브릭)이 2/3으로 부당 감점**되는 회귀 발생. 결론: 2.5-flash-lite에서는 이 실패 유형을 프롬프트만으로 교정할 수 없음 — 지시는 무시되고, 강제 수단은 엄격함이 무관한 루브릭(특히 OR형)으로 번져 완전한 답안에 대한 허위 감점을 낳는다. 허위 감점은 과다 관대화보다 사용자 신뢰에 더 해로우므로 전부 원복했고, 원복 후 `--partial` 재실행으로 기준 상태 복귀 확인(알려진 3건 FAIL, 통제 루브릭 전부 만점). 3.1-flash-lite로 전환한 지금 이 프롬프트 실험을 다시 해봐야 하는지는 미검증.
  - **프롬프트 교정 실험 5차 — 부분 성공, 유지** (2026-07-21, gemini-3.1-flash-lite 기준): 시스템 프롬프트에 규칙 14·15를 추가했다. 2026-07-12의 4차례 실패와 다른 새 지렛대는 **"루브릭 설명의 나열"이 아니라 "모범답안(example_answer_text)"을 필수 제약의 기준선으로 삼은 것**이다 — 모범답안은 AND형 필수 제약("5년 이내", "체감률", "2년 이내")은 명시하고, OR형 선택지는 "등"으로 줄여 하나만 든다. 그래서 "모범답안에 있는 제약이 답안에 없으면 partially_met"이라는 규칙 14는 AND형 누락은 잡으면서 OR형(수증자 요건)은 건드리지 않는다. 규칙 15는 규칙 14의 부작용(틀린-방향 답안에 부분점수를 주어 문제51 결론역전 게이트가 33%→53%로 뚫린 회귀)을 막는다: "결론·주장·포섭의 방향이 모범답안과 정반대면 주제어가 겹쳐도 unmet".
    - **결과**: 문제9 "공급가액 산정방법"의 과다 관대화는 **수정됨**(안정적, 3회 확인 — met→partially_met 2/4). **채점 회귀는 없음**(strong·ambiguous·merged 전부 통과, 문제51 결론역전 게이트 4/15로 오히려 더 견고). `--all` 재실행 시 예상 밖 실패 0건.
    - **한계**: 문제1 물음1("5년 이내" 누락)·물음2·3("판례 법리")과 문제46("선이전 후양도 2년 이내" 누락)은 **여전히 미해결**. 공통점은 답안이 **다른 요소를 길고 풍부하게 나열**해서 단일 제약 누락이 그 풍부함에 묻히는 케이스라는 것(문제9는 답안이 짧고 나열이 적어 규칙 14가 잡았다). 즉 규칙 14는 "답안이 간결할 때"는 통하지만 "답안이 화려하게 길 때"의 누락은 여전히 못 잡는다.
    - **비용**: 규칙 14·15가 모든 채점 호출(물음별 1차 + 교정 재시도)에 ~600자 추가된다. 문제9 한 케이스 수정 대비 프롬프트 상시 증가·행동 변화 리스크를 감수하는 트레이드오프이며, 부분 성공이지만 회귀가 없어 유지하기로 결정. (되돌리려면 `gradeProblem.ts`의 규칙 14·15 두 줄만 제거하면 됨 — 다른 로직과 얽혀 있지 않음.)
  - 남은 선택지: (a) 현재 상태 유지(규칙 14·15로 문제9만 수정, 문제1·46은 알려진 한계로 수용), (b) 문제1·46의 "긴 답안 속 누락"까지 잡는 추가 프롬프트/구조 실험(단, 과도엄격 회귀 리스크), (c) 프롬프트 외 수단(예: met 판정에 대한 완결성 재검증 2차 호출 — 비용 추가) 설계. 현재는 (a) 상태.

## 알려진 한계 (버그 아님, 의도적으로 손대지 않음)

- **근거 인용의 관련성은 검증하지 않음**: `evidenceQuote`가 답안에 실제로 존재하는지(substring)만 확인하고, 그 인용문이 해당 루브릭과 진짜 관련 있는지는 확인하지 않는다. 모델이 무관한 문장을 그럴듯하게 골라 인용하면 통과할 수 있음 — 결정론적 계층으로 완전히 막을 수 있는 종류가 아니라 수용한 잔여 리스크.
- **문제1 물음1 "재산가치 증가사유 발생 요건"이 내용을 다 담았는데도 partially_met(2/3)로 나온 사례**: 압축된 한 문장에 모든 요소를 나열한 것을 "간략하다"고 강도 있게 판단한 경우로, 명백히 틀렸다고 보기 애매한 강도 판단이라 별도로 조정하지 않음.
- **물음 단위 독립 호출 전환으로 API 호출 수가 늘어남**: 문제당 1회 → 물음 개수만큼(보통 2~3회) 병렬 호출 + 총평 합성 1회. 비용 영향은 무시할 만하지만(위 `--partial` 절 참고), Gemini가 일시 과부하(503)일 때 전체 요청이 실패할 확률은 호출 수에 비례해 커진다 — 기존 재시도 로직(`generateContentWithRetry`, 버그#2)이 호출마다 독립적으로 적용되므로 완화되긴 하지만, 완전히 없어지진 않는다.
- **`--ambiguous` 문제1 물음3 "판례 법리"가 ~90% 확률로 force-zero(0/4) — 완곡체 과도엄격 (2026-07-15 수용)**: 답안이 법리를 완곡하게 *맞게* 서술했고 모델도 처음엔 met으로 보지만, `evidenceQuote`를 뽑을 때 canonical 형식 표현("실질적 인과관계가 인정된다면 … 증여세 과세대상에 포함된다")으로 재작성 → 완곡 원문에 그 문장이 없어 `findPhantomEvidence`가 evidence_not_in_answer로 잡고 → 교정 재시도도 형식화 → force-zero. 10회 반복 중 완주한 8회 전부(그리고 이전 관측 합산 시 약 9/10) 0/4. **따라서 `FAIL: 물음 3 "판례 법리" 완곡한 표현이어도 0점 처리되지 않음 (0/4)`은 `--ambiguous`에서 상시 뜨는 알려진 실패이며, 회귀 신호가 아니다.** 좁은 케이스임(같은 답안의 물음2 판례법리 3/3, 물음3 결론·포섭은 완곡체로도 verbatim 통과; 실제 세무사 답안은 형식 법률용어라 매칭됨). 전역 verbatim 검증을 느슨하게 하면 유령근거(버그#4)가 되살아나므로 손대지 않는다. 프롬프트 수정은 위 "미해결로 남은 이슈" 및 `tests/test.md`에서 반복 실패로 문서화됨. 하네스 단정을 0/4로 뒤집어 "무음 처리"하지 않은 이유: 모델 행동이 개선되면 그때 다시 red가 떠야 함.
- **`hasActionableRevisionInstruction`의 허위 음성 2건 (2026-07-21, Slice 1 배포 후 회귀 리뷰에서 수용)**: `--problem51 --overgeneralized` 물음1과 `--overgeneralized`(문제1) 물음2에서 `FAIL: 물음 N 피드백 구체성...`이 뜬다. 두 경우 모두 실제 피드백은 특정 법리 요소("왜 신규성을 갖는지", "개발사업의 요건(구역 지정 등)")를 지목하고 있어 완전히 추상적이진 않지만, 마무리 동사가 "재검토하시기 바랍니다"/"학습하여... 기술하시기 바랍니다"처럼 이 체크가 인식하는 동사 목록(추가/수정/명시/구체화/보완/작성/제시/서술/설명)에 없는 단어라 못 잡는다. `재검토`·`학습`·`이해`·`파악`·`분석`을 한때 목록에 추가했었으나, "판례 법리에 대한 이해가 부족하여 재검토가 필요합니다"처럼 아무것도 지목하지 않는 순수 추상 문장까지 통과시키는 허위 양성이 실측으로 확인되어 되돌렸다(`tests/verify-grading.ts`의 `hasActionableRevisionInstruction` 주석 참고). 허위 양성(추상적 피드백을 놓침)보다 허위 음성(구체적인데 FAIL로 표시됨)이 더 안전한 실패 방향이라는 판단하에 수용 — 이 두 FAIL은 상시 알려진 실패이며 회귀 신호가 아니다.
