# 채점 로직 검증 기록 (문제51 제외)

문제 51은 내용이 곧 변경될 예정이라, 문제51에 특정된 검증(픽스처, 진단 스크립트)은 이 문서와 `verify-grading.ts`에서 모두 제외했다. 다만 아래 버그 A·B는 문제51의 실제 프로덕션 채점 로그에서 처음 발견되었고, 그 발견을 계기로 코드를 고친 뒤 문제1·문제2의 실제 프로덕션 로그로 검증했다 — 수정 자체는 문제51에 한정되지 않는 일반 로직이다.

## 오늘 발견·수정된 채점 로직 버그 2건 (`gradeProblem.ts`)

두 버그 모두 같은 계열: 채점 모델이 `evidenceQuote`(근거 인용문)를 답안 원문 그대로 옮기지 않고 살짝 다르게 재구성해서 인용할 때, 근거 검증 로직(`findPhantomEvidence`/`findSuppressedEvidence`)이 이를 "답안에 없는 근거(유령 근거)"로 오판해 정답인 루브릭까지 강제로 0점 처리하는 문제.

| # | 증상 | 원인 | 수정 |
|---|---|---|---|
| A | 정답 항목 2개 사이에 오답 항목이 하나 끼어 있는 답안에서, 모델이 오답 항목만 건너뛰고 두 정답 문장을 줄바꿈으로 이어붙여 인용 → 답안엔 그 사이에 오답 항목 문장이 실제로 있으니 "연속된 하나의 문자열"로 대조하면 답안에 없는 것으로 판정되어 강제 0점 | `findPhantomEvidence`가 evidenceQuote 전체를 하나의 연속 문자열로만 답안과 대조 | `isEvidenceQuoteVerified` 추가: evidenceQuote를 줄바꿈 단위로 분할해 각 조각을 개별적으로 답안에 실존하는지 검증 (조각마다 여전히 실존해야 하므로 할루시네이션 방지 효과는 유지) |
| B | (A 수정 후에도 잔존) 답안의 인접한 두 줄("1. …\n2. …")을 인용하면서 모델이 번호("2.")를 지우고 공백 하나로 자연스럽게 이어붙임 → 답안 쪽엔 "2."라는 글자가 그 자리에 남아있어 대조가 그 지점에서만 어긋나 마찬가지로 강제 0점 | 줄바꿈 분할만으로는 "번호를 지우고 한 줄로 합친" 경우를 못 잡음 | `stripListMarkers` 추가: 줄 앞의 번호 매기기 기호(`1.` / `(1)` / `①` / `가.` 등)를 구조적 표지로 보고 답안·인용문 양쪽에서 제거한 뒤 대조. 문장 중간의 금액·연도 등 실제 숫자는 건드리지 않도록 줄 시작 위치의 기호만 제거 |

두 수정 모두 `tests/verify-grading.ts`의 독립 재구현 로직(공통 어서션 "근거 인용이 답안에 실존함")에도 동일하게 반영했다.

### 검증 (실제 프로덕션 로그 재실행)

- **문제1 물음1 "수증자 및 재산 취득사유 요건"** (프로덕션 로그 id `0c9a17d5`): 학생 답안이 이 요건을 정확히 담고 있었는데도 원래는 0점 처리됨. 버그 A만 고쳤을 때는 여전히 0점(버그 B 패턴으로 실패), 버그 B까지 고친 뒤 재실행하니 **3/3(met)**으로 정상화, 물음1 총점 6/9 → **9/9**.
- **문제2 물음2** (프로덕션 로그 id `c559ec91`): 답안이 문장 중간에 실제로 잘려 있던(미완성) 경우. 수정 후에도 5개 루브릭 전부 정당하게 **0점 유지** — 수정이 과도한 관대화로 이어지지 않았음을 확인.
- 문제51 회귀(strong 모드, 픽스처 제거 전 마지막 확인): 19/23으로 수정 전후 동일 — 별개로 남아있는 물음3 "처분의 적법성" 이슈(모델이 답안에 있는 결론 문장을 놓친 순수 판단 미스, evidenceQuote 자체가 비어 있어 이번 수정과 무관)는 코드 버그가 아니라 문제51 재작성 시 함께 고려할 사항으로만 남겨둠.

## 문제46 픽스처 확장 (`verify-grading.ts`)

기존에는 문제46이 strong/incomplete 두 모드만 있었고, half/ambiguous/overgeneralized가 없었다. 나머지 문제(1, 9)와 동일한 패턴으로 3개 모드를 추가하고 실제 Gemini 호출로 검증했다.

- **`--half`**: 전 항목 PASS. 총점 14/25(56%) — 물음마다 뒤/가운데/앞 루브릭을 번갈아 비워 위치 편향 없이 확인.
- **`--overgeneralized`**: 전 항목 PASS. 총점 0/25 — 수도권 안/밖 구분, 투자·인원 요건, 감면기간·비율 산정방식을 무시한 자신감 있는 오답에 흔들리지 않고 정확히 0점 처리. 유령 근거 대칭 검증(`findSuppressedEvidence`)도 정상 동작해, "unmet인데 근거 인용됨" 상태를 재검토 후 0점으로 유지.
- **`--ambiguous`**: 총점 22/25(88%)로 임계치(60% 이상)는 통과하지만, 개별 체크 1건 FAIL — 아래 "잔여 이슈" 참고.

## 잔여 이슈 (코드 버그 아님, 별도 조정 안 함)

- **문제46 `--ambiguous` 물음3 "감면대상 소득비율 산출"**: 답안에 "40 나누기 60, 즉 대략 66.6퍼센트(또는 3분의 2)"라는 산출식과 결과값이 명확히 있는데도 모델이 "누락되었다"며 0/3(unmet) 처리. evidenceQuote 자체가 비어 있어 이번에 고친 두 버그(A/B)와는 무관 — 모델이 완곡하게 풀어쓴 문장 속 수치를 그냥 못 알아본, 순수 LLM 인식 실패로 보인다. 재현성이 있다면 향후 조사 대상.
- **문제5(미완성 건축물 취득시기와 특례세율) 물음2 "사용승인일 취득 긍정 근거" 1/3점 사례**: 사용자가 억울한 감점으로 의심해 Supabase에서 실제 `description_display`를 직접 확인한 결과, 루브릭이 "원시취득"이라는 법적 개념을 명시하도록 요구하고 있었고 학생 답안은 이를 언급하지 않아 정당한 부분점수로 결론. 코드 변경 없음 (관련 메모리: `project_grading_strictness_hold`).
