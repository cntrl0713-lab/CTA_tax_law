## 오늘 발견·수정된 채점 로직 버그 2건 (`gradeProblem.ts`)

두 버그 모두 같은 계열: 채점 모델이 `evidenceQuote`(근거 인용문)를 답안 원문 그대로 옮기지 않고 살짝 다르게 재구성해서 인용할 때, 근거 검증 로직(`findPhantomEvidence`/`findSuppressedEvidence`)이 이를 "답안에 없는 근거(유령 근거)"로 오판해 정답인 루브릭까지 강제로 0점 처리하는 문제.

| # | 증상 | 원인 | 수정 |
|---|---|---|---|
| A | 정답 항목 2개 사이에 오답 항목이 하나 끼어 있는 답안에서, 모델이 오답 항목만 건너뛰고 두 정답 문장을 줄바꿈으로 이어붙여 인용 → 답안엔 그 사이에 오답 항목 문장이 실제로 있으니 "연속된 하나의 문자열"로 대조하면 답안에 없는 것으로 판정되어 강제 0점 | `findPhantomEvidence`가 evidenceQuote 전체를 하나의 연속 문자열로만 답안과 대조 | `isEvidenceQuoteVerified` 추가: evidenceQuote를 줄바꿈 단위로 분할해 각 조각을 개별적으로 답안에 실존하는지 검증 (조각마다 여전히 실존해야 하므로 할루시네이션 방지 효과는 유지) |

### 되돌린 것 — `ThinkingLevel.MINIMAL` 리터럴 교체 시도

검토 중 `ThinkingLevel.MINIMAL`이 런타임에서 `"MINIMAL"`(대문자)로 해석되어 공식 문서의 `thinkingLevel` 타입 예시(소문자 `"minimal"`)와 다르다는 점을 지적하고 리터럴 `'minimal'`로 교체를 시도했으나, **`tsc`가 즉시 타입 에러를 냈다** — 이 SDK의 `GenerateContentConfig`가 실제로 참조하는 `ThinkingConfig.thinkingLevel`은 `ThinkingLevel` enum 타입이라 소문자 문자열 리터럴이 아니라 enum 값이 타입상 올바르다(컴파일러가 직접 "`ThinkingLevel.MINIMAL`을 쓰려던 것 아니냐"고 제안함). 원래 구현이 맞았고, 리터럴 교체 시도는 되돌렸다 — API 자체는 대문자/소문자를 구분하지 않는 것으로 실측되었지만(별도 스크립트로 확인, thoughts=0/output/total 동일), 타입 정합성 관점에서는 enum이 정답이다.

### `--all` 종료 코드 신뢰성 개선 (부수 작업)

"피드백 구체성" 체크는 모델 피드백 어미에 따라 물음·모드를 가리지 않고 들쭉날쭉 실패하는 문서화된 트레이드오프(허위 음성)라, 특정 (문제,모드,물음)을 `KNOWN_EXCEPTIONS`에 하나하나 등록하는 방식이 무의미했다(등록 안 한 곳에서 또 떠서 `--all` 종료 코드가 오염됨). `runAll()`에서 이 체크의 실패는 fixture/mode 무관하게 일괄 "알려진 실패"로 분류하도록 변경 — 체크는 계속 실행·출력되지만 종료 코드는 진짜 채점 회귀에만 반응한다. 또한 규칙 14로 수정된 문제9 partial "공급가액 산정방법"은 `KNOWN_EXCEPTIONS`에서 뺐다(다시 met으로 회귀하면 "예상 밖 실패"로 알림이 가도록 의도적으로 미등록).
